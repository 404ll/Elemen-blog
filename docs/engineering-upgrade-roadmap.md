# Blog 工程化升级学习文档

> 目标：把当前站点从“作品级”升级为“工程级技术博客”。

## 1. 当前阶段定位

- 优势：
  - Next.js App Router + MDX 内容体系
  - 已有主题切换、双语、代码高亮、阅读增强（目录/进度）
- 短板：
  - 内容校验不完整（frontmatter 强约束不足）
  - 自动化质量门禁薄弱（测试/CI）
  - 线上可观测性弱（错误/性能）

---

## 2. 升级原则（学习重点）

1. **先可验证，再可扩展**：先补校验与测试，再叠加新功能。
2. **先质量闭环，再功能炫技**：构建失败要能阻断问题进入线上。
3. **每次只做一层能力**：避免同时改内容系统 + 构建链路 + UI。

---

## 3. 分阶段路线

## Phase A：内容工程化（P0）

### 目标
让文章内容具备“结构化输入 + 构建时校验 + 可预测输出”。

### 任务
- [ ] 引入 `gray-matter` 解析 frontmatter
- [ ] 引入 `zod` 定义 `PostSchema`
- [ ] 统一字段：`title/date/excerpt/category/tags/cover/draft/updatedAt`
- [ ] 对非法 frontmatter 在构建阶段直接报错
- [ ] 增加 `content/example.mdx` 作为规范模板

### 学到什么
- schema 驱动内容系统
- 构建期失败策略
- 内容与渲染解耦

---

## Phase B：质量门禁（P0）

### 目标
形成“本地可验证 + CI 可阻断”的最小闭环。

### 任务
- [ ] 增加脚本：`typecheck` / `lint` / `test`
- [ ] 新增 GitHub Actions：PR 必跑检查
- [ ] 关键逻辑加单测（至少覆盖 `lib/post.ts` 的解析与排序）
- [ ] 为阅读增强逻辑加基础行为测试（可后置）

### 学到什么
- CI 作为质量边界
- 回归成本控制
- 小项目测试策略

---

## Phase C：可观测与性能（P1）

### 目标
出问题可发现，性能可度量，优化可验证。

### 任务
- [ ] 接入错误监控（如 Sentry）
- [ ] 接入访问分析（如 Umami/GA）
- [ ] 建立 Web Vitals 观察基线
- [ ] 图片策略统一（尺寸、懒加载、封面约束）

### 学到什么
- 从“感知问题”到“量化问题”
- 前端性能预算思维
- 线上问题定位流程

---

## Phase D：交付与规范（P1）

### 目标
让项目具备稳定迭代能力。

### 任务
- [ ] PR 模板（变更说明/测试项）
- [ ] issue 模板（Bug/Feature）
- [ ] release note 习惯化
- [ ] 文档化常见操作（新建文章、发布前检查）

### 学到什么
- 工程协作基本盘
- 变更可追踪
- 文档驱动开发

---

## 4. 每周节奏建议

- Week 1：Phase A（内容工程化）
- Week 2：Phase B（质量门禁）
- Week 3：Phase C（监控与性能）
- Week 4：Phase D（交付规范）

> 每周结束产出：
> - 一份“本周改了什么”
> - 一份“踩坑与复盘”
> - 一条“下周最小目标”

---

## 5. 完成标准（Definition of Done）

- 新文章 frontmatter 不合法时能被自动拦截
- PR 不通过 lint/typecheck/test 不能合并
- 线上错误可追踪，性能有可读数据
- 新写作者（未来的你）能按文档快速上手
