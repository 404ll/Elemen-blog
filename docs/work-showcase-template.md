# 工作展示模板

用于把一个项目文件、README 或粗略大纲整理进 `/practice/work`。

默认不追问展示对象和展示形式，先按“博客读者能看懂、面试时也能复用”的角度输出一版。

## 使用方式

把项目材料贴给 Codex，然后说：

```txt
按 docs/work-showcase-template.md 帮我整理成一个 /practice/work 展示页。
```

如果没有明确指定展示角度，默认按下面三点组织：

- 面向读者：先讲清楚项目解决了什么问题。
- 面向面试：突出个人贡献、技术判断和取舍。
- 面向作品集：保留可以公开展示的流程、结构和代码片段。

## 输入

```txt
项目文件 / README / 仓库链接
我想表达的大纲或重点
```

输入材料可以很粗糙，不需要先整理成文章。可以包含：

- README、设计稿说明、任务描述。
- 关键代码文件或目录结构。
- 自己做过的部分、踩过的坑、复盘想法。
- 不方便公开的信息说明，例如真实项目名、客户名、接口、域名、token。

## 输出结构

```md
# 项目名

## 场景

说明项目解决的问题、为什么要做，以及它不做什么。

## 我做了什么

用 3-5 条写清楚主要职责和贡献。

## 核心流程

用文本图或 Mermaid 图讲清楚主链路。

## 数据流

说明关键数据结构怎么流转。

## 模块拆分

说明主要模块职责，突出边界和可维护性。

## 技术难点

挑真正值得展示的 2-4 个点，不堆实现细节。

## 代码

放核心代码片段、伪代码或入口流程。

## 记一下

写复盘：这个项目最值得记住的工程判断是什么。
```

## 输出文件

每个展示页使用一个独立目录：

```txt
practice/work/<id>/
  article.md
  code.js
```

- `article.md` 放完整展示内容。
- `code.js` 只放短的入口流程、核心片段或伪代码。
- `<id>` 使用英文小写和短横线，例如 `seo-slug-normalize`。
- 如果项目没有适合公开的代码，可以省略 `code.js`，但文章里要保留流程图或伪代码。

## 生成原则

- 优先提炼主线，不复述 README。
- 先讲问题和链路，再讲模块和代码。
- 复杂项目必须补流程图或数据流图。
- API、token、私有域名默认 mock 或脱敏。
- 不展示真实客户名、内部项目名、私有仓库名、私有接口地址。
- 不把“我参与了”写得很虚，必须落到具体模块、判断或结果。
- 不堆技术名词，技术点要服务于项目问题。
- 文章要能独立阅读，不能假设读者看过原始 README。

## 脱敏规则

- 真实项目名改成中性标题，例如“搜索路由规范化工具”。
- 公司、客户、团队、供应商名称用“业务方”“内部系统”“第三方服务”替代。
- 域名、接口、token、账号、ID、工单号全部 mock。
- 业务数据只保留量级和形态，不保留真实值。
- 代码片段只保留关键逻辑，不复制大段生产代码。

## 完成标准

- 读者能在 1 分钟内知道项目解决什么问题。
- 面试时能直接用“场景 - 贡献 - 难点 - 结果”讲出来。
- 没有泄露真实项目名、私有路径、账号、接口或业务数据。
- `article.md` 和 `code.js` 的内容互相对应。
